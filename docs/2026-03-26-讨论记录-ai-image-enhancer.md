## 一、技术架构决策

### 1.1 不需要 Python 后端，纯 Next.js 搞定

**结论**：MVP 用 Next.js + Vercel Edge + Replicate API + Supabase，全 Serverless，不用 Python。

**原因**：
- AI 推理交给 Replicate API，不需要本地跑 GPU
- 鹿班只掌握 Next.js + TypeScript，不用同时维护 Python 服务，开发速度快一倍
- 图片上传、存储走 R2，限流走 Supabase，API Routes 搞定一切

**架构**：
```
用户 → Next.js (SSR + API Routes)
         ├── 文件上传 → Cloudflare R2
         ├── AI 推理 → Replicate API（Webhook 回调）
         ├── 限流 → Supabase (PostgreSQL)
         └── 结果返回 → CDN → 用户下载
```

### 1.2 前期不需要部署开源模型

**结论**：MVP 全部用 Replicate API，按量付费。

**何时自部署**：月处理量超过 5 万张时，自部署 RunPod A100（~$300/月）更划算。

## 二、去背景方案

**结论**：用 **RMBG-2.0**（开源）走 Replicate API 托管，不需要 Remove.bg。

| 方案 | 效果 | 成本 |
|------|------|------|
| RMBG-2.0（BRIA） | ⭐⭐⭐⭐⭐ | Replicate ~$0.002-0.01/张 |
| Remove.bg API | ⭐⭐⭐⭐ | $0.10-0.50/张（贵 50 倍） |

效果一样好，成本低 50 倍，没有理由用 Remove.bg。

## 三、数据库选择

**结论**：用 Supabase，不用 D1。

| 维度 | Supabase | D1 |
|------|----------|-----|
| Auth | ✅ 一体化 | ❌ 要另接 |
| 免费额度 | 500MB 数据库 | 5GB 存储 |
| 好用度 | Dashboard 完善 | 简陋 |

D1 的唯一优势是 Cloudflare 生态统一，但 Auth 缺失是硬伤。Supabase 的 Auth + DB 一站式方案，省掉大量开发工作。

## 四、定价策略讨论

### 4.1 为什么从 $7 调整为 $9.9？

**原始定价 $7 的问题**：
- 离成本太近（Starter 1500 张/月，Replicate 成本约 $7.5-15），扣掉支付手续费（3%），毛利薄
- 没有数据支撑，是拍脑袋定价

**调到 $9.9 的理由**：
- 竞品 Let's Enhance 是 $9/月，我们卖 $9.9 但"无水印+无注册"体验更好，物超所值
- 结尾 .9 是经典定价策略，用户感知 <$10，实际多赚 $0.9/用户
- 按 1000 付费用户算，$9.9 vs $9 每月多赚 $900，一年多赚 $10,800

### 4.2 免费额度够用 / 付费驱动力

**核心问题**：无注册 5 张/日 + 注册 10 张/日，用户还会付费吗？

**分析**：

| 用户类型 | 免费够吗？ | 会付费吗？ |
|---------|----------|----------|
| 偶尔修图个人用户 | ✅ 够 | ❌ 不会（本来就不会付费） |
| 社交媒体发图 | ✅ 够 | ❌ 不会 |
| 小电商卖家（每天上新几十张） | ❌ 不够 | ✅ 会 |
| 内容创作者（每周出大量图） | ❌ 不够 | ✅ 会 |
| 房产经纪人（批量处理） | ❌ 不够 | ✅ 会 |
| 需要 4K/8K 放大 | ❌ 不够（免费 2x） | ✅ 会 |

**付费驱动力（转化钩子）**：
1. **分辨率差**：免费 2x → 付费 4x/8x，试过 4K 效果后回不去
2. **批量处理**：免费只能单张，批量是刚需
3. **速度差异**：免费排队 vs 付费优先
4. **背景去除**：免费版没有

**结论**：免费用户是流量引擎（帮做 SEO + 口碑），付费靠分辨率差 + 批量需求驱动。不需要担心没人付费。

### 4.3 客户画像修正

PRD 原文写"小电商卖家觉得 Let's Enhance $9/月太贵"——这个逻辑有问题。修正后的付费客户画像：

## 七、技术栈选型讨论（Cloudflare 全家桶）

### 7.1 背景

嗲地明确以后站点主要部署在 Cloudflare，要求技术栈与 Cloudflare 生态配合。

### 7.2 Cloudflare 生态能力评估

| 云服务 | 对应能力 | 匹配度 |
|--------|---------|--------|
| **Pages** | 静态/SSR 托管 | ⭐⭐⭐⭐ |
| **Workers** | Edge 计算（API） | ⭐⭐⭐⭐⭐ |
| **R2** | 对象存储 | ⭐⭐⭐⭐⭐ |
| **D1** | SQLite 数据库 | ⭐⭐⭐⭐ |
| **Queues** | 消息队列 | ⭐⭐⭐⭐⭐ 异步 AI 推理 |
| **Workers AI** | GPU 推理 | ⭐⭐⭐⭐ 可替代 Replicate |
| **Images** | 图片处理/CDN | ⭐⭐⭐⭐⭐ |
| **Hyperdrive** | 数据库代理 | ⭐⭐⭐⭐ |

### 7.3 Workers AI vs Replicate

| 维度 | Replicate | Workers AI |
|------|-----------|-----------|
| 模型选择 | 海量 | 有限但够用 |
| 成本 | $0.005-0.02/张 | ~$0.01/张，有免费额度 |
| 延迟 | 外部 API | 同一生态，更低 |
| 免费额度 | 无 | 有 |

**结论**：Workers AI 主力，Replicate 做备选。

### 7.4 两个方案对比

**方案 A（全面 Cloudflare）**：
- Pages + Workers + R2 + D1 + Queues + Workers AI + Clerk Auth
- 优势：统一生态、运维简单、成本最低、未来可复用
- 成本：起步 ~$10-30/月，规模化 ~$180/月

**方案 B（混合方案）**：
- Pages + Workers + R2 + Queues + Supabase (DB + Auth)
- 优势：Supabase 功能更全，Auth 不用额外接
- 劣势：外部服务多一层延迟，成本更高

### 7.5 决策

**选方案 A**。理由：
1. 嗲地说以后站点主要部署 Cloudflare，统一体系利于复用
2. D1 + Workers 组合其他项目也能用
3. 成本更低（Supabase Pro $25/月 vs D1 免费 5GB）
4. Clerk 免费 10K 用户足够 MVP
5. Queues 异步处理 AI 推理，架构更优雅

### 7.6 D1（SQLite）的局限性

D1 不支持：实时订阅、复杂全文搜索、行级安全策略（RLS）。

**对 MVP 影响**：无。我们的场景就是查额度、记录处理历史、存用户信息，SQLite 完全够用。
